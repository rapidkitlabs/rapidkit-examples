# Example Workspace Publication Contract

This contract defines what a cloneable Workspai example publishes.

## Required workspace files

Every published workspace must commit:

- `README.md`
- `.gitignore`
- `.workspai-workspace`
- `.workspai/workspace.json`
- `.workspai/workspace.contract.json`
- `.workspai/policies.yml`
- `.workspai/toolchain.lock`
- `.workspai/cache-config.yml`

The workspace contract must use relative project paths, contain no secrets or
machine-local paths, and agree with [examples.json](examples.json).

Published Python workspaces pin `rapidkit-core==0.5.5` exactly and commit the
matching Poetry lockfile. Updating that baseline requires module regeneration,
project tests, runtime smoke checks, and this repository gate; changing only
the dependency string is not sufficient.

Every project `registry.json` is the project-level record of installed Core
modules. Snippet registries must not carry unresolved failed or pending entries.
Source wrappers that identify generated Core implementations must select the
same version recorded by the registry. Generated vendor payloads are not part
of the published example source.

## Required user journey

Every published workspace README must help a user start from a fresh clone or
download without relying on machine-local state. It must include:

- `workspace sync` as the first registry and contract reconciliation step
- strict workspace contract verification
- canonical workspace model generation
- a clear statement that complete workspaces are synchronized, not imported or
  adopted as projects
- a link to [WORKSPACE_ONBOARDING.md](WORKSPACE_ONBOARDING.md) for legacy
  foundation migration, create/import/adopt choices, the Workspace Intelligence
  chain, agent grounding, CI, and recovery
- a clear explanation that Workspai complements existing agents, models, IDEs,
  and frameworks by generating evidence-backed context rather than replacing
  those tools

The repository validator enforces these minimums for real examples and raw
profile fixtures. User-facing commands must remain aligned with the current
Workspai CLI contract.

## Required project files

Every runnable project must commit:

- source code and tests
- package/build manifests and reproducible dependency lockfiles
- `.env.example` with placeholders only when environment variables are needed
- Docker/CI configuration when it is part of the documented example
- canonical `.workspai/project.json` for Workspai-created or explicitly adopted
  projects

The current Core-backed examples predate canonical project metadata. Their
`.rapidkit/project.json`, context, launcher, file hashes, and snippet registry
remain compatibility inputs until those projects are regenerated by compatible
Workspai-owned kits. Do not label them as canonical project outputs.

Do not publish `.rapidkit/snapshots/`, `.rapidkit/audit/`,
`.rapidkit/reports/`, `.rapidkit/tmp/`, or `.rapidkit/vendor/`. They are generated
module/runtime state or execution history, not portable project identity. A
fresh clone reconstructs required runtime state through the supported RapidKit
module installation/restore path using the committed registry and locks.

## Conditional files

Publish these only when the example explicitly teaches or consumes them:

- `AGENTS.md`, Skills, and agent instructions generated from the workspace
- infrastructure plans or compose files intended for local development
- sanitized evidence fixtures stored outside live `.workspai/reports/` paths
- legacy `.rapidkit*` metadata required by an older Core-backed project
- compatibility launchers generated when Python Core installation is deferred

Conditional files must be portable, reviewable, and covered by the example.

## Never publish

Do not commit:

- `.workspai/workspace-registry.v1.json`
- `.workspai/imported-projects.json`
- `.workspai/adopt.json` or `.workspai/adopt-readiness.json`
- live `.workspai/reports/`, caches, histories, snapshots, or archives
- generated `.rapidkit/vendor/`, snapshots, audit logs, reports, or temporary files
- absolute filesystem paths or local registry locations
- `.env`, credentials, tokens, private keys, customer evidence, or entitlements
- `.venv`, `node_modules`, build output, coverage output, or IDE-local state

If a report is needed for documentation or tests, copy a sanitized payload to a
clearly named fixture directory and validate it against its schema.

## Reconciliation after clone

```bash
npx workspai workspace sync
npx workspai workspace contract verify --strict --json
npx workspai workspace model --json --write
```

`workspace sync` recreates the machine-local registry summary and reports from
the portable files. Those regenerated artifacts remain uncommitted.

## Change gate

Before publishing:

```bash
corepack npm run check
```

The local gate verifies catalog parity, profile completeness, portable paths,
contracts, ports, Core/module/wrapper alignment, dependency locks, snippet
state, onboarding coverage, local links, legacy command drift, and publication
ignores. CI additionally installs and tests all six Python projects and builds
and tests both NestJS projects.
