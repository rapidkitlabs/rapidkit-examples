# Workspace Onboarding

Use this guide for every workspace in this repository, including raw profile
fixtures and older RapidKit Core examples. Run workspace commands from the
workspace root unless a project README explicitly says otherwise.

## Prerequisites

- Node.js 20.19.6 or newer
- `npx` access to the `workspai` npm package
- The runtime required by the projects you intend to run

Confirm the CLI before changing workspace state:

```bash
npx workspai version --json
npx workspai --help
```

## RapidKit Core compatibility

The runnable examples are validated against `rapidkit-core==0.5.5`. The three
Python workspace roots pin that version exactly. Each project records its own
runtime dependencies in a lockfile and its installed Core modules in
`registry.json`.

Install the workspace environment before running Core-backed project commands:

```bash
cd <workspace-directory>
poetry install --no-interaction
poetry run rapidkit --version
```

Each Core-backed project also owns its runtime lockfile. Install and test it
from the project directory:

```bash
cd <project-directory>
poetry install --no-interaction   # Python project
poetry run pytest -q

npm ci                            # NestJS project
npm run build
npm test -- --runInBand
```

Core module compatibility is not represented by `registry.json` alone. A valid
example requires all of the following to agree:

- the installed version in the project `registry.json`
- the version selected by any source wrapper
- the matching `.rapidkit/vendor/<module>/<version>` snapshot
- an applied snippet registry with no `pending` or `failed` entries

When upgrading Core, update modules through the RapidKit Core command line,
reconcile snippets, regenerate dependency locks, run project tests, and run the
repository gate. Do not hand-edit only the registry version.

## Start after clone or download

Enter the directory containing `.workspai-workspace`:

```bash
cd <workspace-directory>
npx workspai workspace sync
npx workspai workspace contract inspect
npx workspai workspace contract verify --strict --json
npx workspai workspace model --json --write
```

`workspace sync` is the required first reconciliation step. It discovers the
projects visible from this workspace, refreshes the portable workspace
contract, and registers a cloned, downloaded, moved, or previously unregistered
workspace in the machine-local registry. It is safe to run repeatedly.

Confirm local discovery:

```bash
npx workspai workspace list --refresh --json
npx workspai doctor workspace --json
```

Do not commit the global registry or generated `.workspai/reports/*` files.
They describe a specific machine or run and are rebuilt from portable workspace
state.

## Open a legacy workspace

Some older examples retain `.rapidkit-workspace` and `.rapidkit/*` metadata
because their projects were generated by RapidKit Core. If a downloaded
workspace does not yet contain `.workspai-workspace`, migrate its foundation
before synchronization:

```bash
cd <legacy-workspace>
npx workspai workspace foundation ensure
npx workspai workspace sync
npx workspai workspace contract verify --strict --json
npx workspai workspace model --json --write
```

Keep legacy project metadata when the project still depends on it. The
canonical public workspace interface is `.workspai-workspace` plus
`.workspai/*`.

## Add projects to a workspace

### Create a supported project

Create places a new project under the current workspace:

```bash
cd <workspace-directory>
npx workspai create project
# Or choose a known kit directly:
npx workspai create project nextjs web --yes
npx workspai create project fastapi.standard api --yes
npx workspai workspace sync
```

### Import a project or repository

Use import when the source should be copied or cloned into the workspace:

```bash
cd <workspace-directory>
npx workspai import ../existing-project --workspace . --json
npx workspai import https://github.com/acme/service.git --git --workspace . --json
npx workspai workspace sync
```

### Adopt a project in place

Use adopt when the source must remain at its current path:

```bash
cd <workspace-directory>
npx workspai adopt ../existing-project --workspace . --dry-run --json
npx workspai adopt ../existing-project --workspace . --json
npx workspai workspace sync
```

Import changes where the project lives. Adopt does not move or copy source.
Both routes make the project observable to Workspace Intelligence. Only use
`--enable-modules` when existing RapidKit metadata proves that Core module
commands are supported.

## Reconcile an existing workspace repository

Do not import or adopt a directory that is already a complete workspace with
`.workspai-workspace` and `.workspai/workspace.json`. Clone or download it,
enter its root, and run:

```bash
npx workspai workspace sync
npx workspai workspace contract verify --strict --json
npx workspai workspace model --json --write
```

The portable markers establish workspace identity. The first sync repairs the
missing machine-local registry entry.

## Establish the first intelligence baseline

After projects have been created, imported, or adopted, create a stable model
and comparison baseline:

```bash
npx workspai workspace model --json --write
npx workspai workspace snapshot --json
```

The snapshot is the baseline for later structural diffs. Commit source and
portable workspace contracts according to your repository policy; generated
reports remain runtime evidence.

## Run the Workspace Intelligence chain

After source, dependency, policy, ownership, or contract changes, compare the
current model with the snapshot created before those changes. Do not overwrite
that baseline before calculating the diff:

```bash
# 1. Current software-system model
npx workspai workspace model --json --write

# 2-3. Structural change against the existing stable baseline
npx workspai workspace diff \
  --from .workspai/reports/workspace-model-snapshot.json --json

# 4. Transitive blast radius
npx workspai workspace impact \
  --from .workspai/reports/workspace-model-diff-last-run.json --json

# 5-8. Current health, analysis, contract, and release evidence
npx workspai doctor workspace --json
npx workspai analyze --json --strict
npx workspai workspace contract verify --strict --json
npx workspai readiness --json

# 9. Definitive workspace gate
npx workspai workspace verify \
  --from-impact .workspai/reports/workspace-impact-last-run.json --strict --json

# 10-11. Agent context and portable agent/IDE surfaces
npx workspai workspace context --for-agent --json --write --no-agent-sync
npx workspai workspace agent-sync --write --json --preset enterprise

# 12. Evidence-backed explanation
npx workspai workspace explain release-blocked --json --write
```

After the change has passed review and verification, intentionally advance the
baseline for the next change cycle:

```bash
npx workspai workspace snapshot --json
```

Do not present an old report as current evidence. If source or workspace inputs
change, rerun the producing step and every dependent downstream step.

## What the chain produces

| Need | Canonical output |
| --- | --- |
| System structure and graph | `.workspai/reports/workspace-model.json` |
| Comparison baseline | `.workspai/reports/workspace-model-snapshot.json` |
| Structural change | `.workspai/reports/workspace-model-diff-last-run.json` |
| Blast radius | `.workspai/reports/workspace-impact-last-run.json` |
| Health and contract evidence | Doctor and contract verification reports |
| Release verdict | `.workspai/reports/workspace-verify-last-run.json` |
| Agent context | `.workspai/reports/workspace-context-agent.json` |
| Agent and IDE grounding | `AGENTS.md`, skills, `INDEX.json`, customization pack |
| Human-readable reasoning | `.workspai/reports/workspace-explain-last-run.json` |

Developers, CI, IDEs, and AI agents should consume these structured artifacts
instead of parsing terminal prose.

## Generate files for AI agents and IDEs

Workspai does not replace an agent, model, IDE, or coding assistant. It prepares
one evidence-backed workspace context and projects it into portable files that
different consumers can read.

Generate context first, then distribute it:

```bash
npx workspai workspace context --for-agent --json --write --no-agent-sync
npx workspai workspace agent-sync --write --json --preset enterprise
```

The enterprise preset generates or refreshes the canonical agent inventory,
including:

- `.workspai/reports/workspace-context-agent.json` for compact structured context
- `.workspai/reports/INDEX.json` for artifact discovery and read order
- `.workspai/reports/agent-customization-pack.json` for the generated inventory
- `.workspai/reports/workspace-skills-index.json` and `.workspai/skills/*.md`
- `AGENTS.md` and `.workspai/AGENT-GROUNDING.md` as portable entry points
- supported Copilot, Cursor, Claude, Codex, and VS Code instruction surfaces

To generate only selected ecosystems, provide a target list:

```bash
npx workspai workspace agent-sync \
  --write --json --preset enterprise --target copilot,cursor,claude,codex
```

These files do not create a second source of truth. They are generated views of
the same workspace model, contracts, impact, freshness, and verification
evidence used by developers and CI. After meaningful workspace changes, rerun
the upstream intelligence steps before refreshing context and agent surfaces.

Review generated instructions like any other project artifact. Commit portable
agent files only when your repository policy wants stable team-wide grounding;
keep machine-local reports and absolute registry data uncommitted.

## Initialize and run projects

Run project commands from an individual project directory:

```bash
cd <project-directory>
npx workspai init
npx workspai test
npx workspai build
npx workspai dev
```

For all discovered projects, run a fleet stage from the workspace root:

```bash
npx workspai workspace run init
npx workspai workspace run test --json
npx workspai workspace run build --json
```

## Before CI or release

```bash
npx workspai pipeline --json --strict
```

Use strict JSON commands in CI so verdicts are carried by structured output and
exit codes rather than human-readable text.

## Recovery checks

```bash
npx workspai workspace sync
npx workspai workspace contract inspect
npx workspai workspace contract verify --strict --json
npx workspai doctor workspace --json
npx workspai workspace model --json --write
```

If the CLI cannot find the workspace, verify that you are inside the directory
containing `.workspai-workspace`. For an old workspace with only legacy
markers, run `workspace foundation ensure` first.
